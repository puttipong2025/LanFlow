# ระบบ Authentication ของ LanFlow 🔐

เอกสารนี้สรุปการทำงาน โครงสร้างสถาปัตยกรรม (Architecture) จุดที่ควรพัฒนาต่อ และจุดเสี่ยงที่ต้องระวังเรียงตามลำดับความสำคัญ สำหรับระบบล็อกอินและการยืนยันตัวตนในปัจจุบันของ LanFlow

---

## 1. โครงสร้างและการทำงานแบบละเอียด (Architecture Workflow)

ระบบ Authen ของโปรเจกต์นี้ **ไม่ได้ใช้ Supabase Auth** แบบ Native (ไม่ได้ใช้ระบบล็อกอินที่แถมมากับ Supabase) แต่เป็นการเขียน **Custom JWT (JSON Web Token)** ขึ้นมาเอง โดยใช้ตาราง `profiles` บน Supabase เป็นฐานข้อมูลผู้ใช้ธรรมดา

### 1.1 กระบวนการล็อกอิน (Login Flow)
1. **Frontend:** ผู้ใช้กรอกเบอร์โทรศัพท์และรหัสผ่าน ส่ง POST request ไปที่ `/api/auth/login`
2. **Backend (`login/route.ts`):** 
   - ระบบใช้ `SUPABASE_SERVICE_ROLE_KEY` ดึงข้อมูลผู้ใช้จากตาราง `profiles` (ข้ามสิทธิ์ RLS เพราะยังไม่ได้ล็อกอิน)
   - เช็คว่ารหัสผ่านที่กรอกมา ตรงกับ `password_hash` ในฐานข้อมูลหรือไม่ ผ่านไลบรารี `bcryptjs`
   - หากถูกต้อง ระบบจะค้นหาสาขา (Location) ของผู้ใช้จากตาราง `user_locations`
3. **การสร้าง Token (`src/lib/server/auth.ts`):**
   - ระบบใช้ไลบรารี `jose` สร้าง JWT โดยอัดข้อมูลสำคัญ (Payload) ลงไป ได้แก่ `sub` (User ID), `phone`, `role`, และ `locationIds`
   - Token มีอายุการใช้งาน 7 วัน (รองรับการใช้งานออฟไลน์ของ PWA)
4. **การเก็บ Token (`use-auth.ts`):**
   - หน้าบ้านเมื่อได้ Token จะเก็บไว้ 2 ที่:
     1. `localStorage` (เอาไว้ใช้แนบ Header `Bearer <token>` เวลาเรียก API)
     2. `Cookie` (ชื่อ `lanflow-token` เอาไว้รองรับ Middleware ในอนาคต)

### 1.2 กระบวนการเรียกใช้ API (API Authentication)
1. **Frontend:** ทุกครั้งที่ fetch ข้อมูลจาก `/api/lanflow/*` จะต้องเรียกผ่านฟังก์ชัน `authFetch` หรือแนบ `Authorization: Bearer <token>`
2. **Backend:** 
   - ทุก API Route จะต้องเรียก `await requireAuth(request)` เพื่อถอดรหัส JWT และตรวจสอบว่าหมดอายุหรือยัง
   - ฟังก์ชันจะคืนค่า `result.auth` ซึ่งประกอบด้วย `sub` (User ID) และสิทธิ์ต่างๆ นำไปใช้บันทึกฐานข้อมูลในฐานะคนทำรายการ (`created_by_user_id`)

## 2. จุดที่ต้องระวัง / สิ่งที่ต้องแก้ (เรียงลำดับความสำคัญ)

จุดเหล่านี้คือช่องโหว่หรือปัญหาที่อาจเกิดขึ้นได้ ควรนำไปวางแผนแก้ไขเป็นลำดับแรกๆ:

> [!CAUTION]
> **Priority 1: ช่องโหว่เรื่อง Data Isolation (การแบ่งแยกข้อมูลข้ามสาขา)**
> ปัจจุบัน Backend API ส่วนใหญ่ใช้ `getAdminClient()` ซึ่งมีสิทธิ์เข้าถึงฐานข้อมูล *ทั้งหมด* (Bypass RLS) 
> - **ความเสี่ยง:** หาก API ของการ แก้ไข (PATCH) หรือลบ (DELETE) ลืมตรวจสอบว่า `locationId` ของเอกสารนั้น ตรงกับ `locationIds` ที่อยู่ใน Token ของพนักงานหรือไม่ พนักงานสาขา A อาจจะบังเอิญ (หรือจงใจ) ลบข้อมูลของสาขา B ได้
> - **วิธีแก้:** ต้องบังคับเช็ค `locationIds.includes(record.location_id)` เสมอในระดับ API ก่อนอัปเดตหรือลบข้อมูล 

> [!WARNING]
> **Priority 2: ไม่มีระบบเปลี่ยนรหัสผ่าน (Password Management)**
> - ปัจจุบันรหัสผ่านถูก Hash ด้วย `bcrypt` และฝังไว้ใน Database ผ่านโค้ด 
> - **ความเสี่ยง:** ยังไม่มี API สำหรับให้ผู้ใช้ "เปลี่ยนรหัสผ่านตัวเอง" และแอดมินก็ไม่มีปุ่ม "รีเซ็ตรหัสผ่าน" ให้พนักงาน หากพนักงานลืมรหัสผ่านจะไม่สามารถเข้าสู่ระบบได้เลยต้องแก้ Database สด
> - **วิธีแก้:** สร้าง API `/api/auth/change-password` และเพิ่ม UI ในเมนูโปรไฟล์

> [!IMPORTANT]
> **Priority 3: การเก็บ JWT ไว้ใน LocalStorage**
> - **ความเสี่ยง:** การเก็บ Token ไว้ใน LocalStorage มีความเสี่ยงต่อการถูกโจมตีแบบ XSS (Cross-Site Scripting) หากมีสคริปต์ประสงค์ร้ายรันบนหน้าเว็บ มันจะขโมย Token ได้
> - **วิธีแก้:** ย้าย Token ไปเก็บใน **HttpOnly Cookie** เท่านั้น (ซึ่งทำให้ JavaScript หน้าบ้านอ่านไม่ได้) และเปลี่ยน API ให้ดึง Token จาก Cookie แทน Header

## 3. จุดที่ควรพัฒนาต่อในอนาคต (Future Improvements)

1. **ระบบ Refresh Token / Auto-extend Session:**
   - ปัจจุบัน Token หมดอายุใน 7 วัน เมื่อถึงวันที่ 8 ผู้ใช้จะถูกเตะออกจากระบบกะทันหัน 
   - **แนวทาง:** ควรใช้ฟังก์ชัน `isTokenExpiringSoon` ที่เขียนเตรียมไว้แล้วใน `auth.ts` เพื่อเช็คว่าถ้า Token ใกล้หมดอายุ (เช่น เหลือ 1 วัน) ให้ API รีเฟรชและส่ง Token ใหม่กลับมาเงียบๆ ผู้ใช้จะได้ไม่ต้องล็อกอินใหม่บ่อยๆ

2. **Route Protection ด้วย Next.js Middleware:**
   - ปัจจุบันการป้องกันคนยังไม่ล็อกอินแอบเข้าหน้าเว็บ ทำโดย React หน้าบ้านเช็คแล้วเด้งกลับไปหน้า `/login` ทำให้หน้าเว็บอาจจะกระพริบโหลดก่อน 1 วินาที
   - **แนวทาง:** เขียนไฟล์ `middleware.ts` ตรวจจับ URL ทันทีตั้งแต่ฝั่ง Server ถ้าไม่มี Cookie `lanflow-token` ให้ Redirect ไป `/login` ทันที จะได้ UX ที่เนียนตาขึ้น

3. **ย้ายไปใช้ Supabase Auth (พิจารณาระยะยาว):**
   - เนื่องจากเราใช้ Supabase อยู่แล้ว การใช้ Custom JWT ทำให้เราสูญเสียความสามารถ **RLS (Row Level Security)** ของ Supabase ไป (เราต้องมาเช็คสิทธิ์ด้วย if-else เองใน API ทุกเส้น)
   - **แนวทาง:** หากระบบสเกลใหญ่ขึ้น การย้ายไปใช้ Supabase Auth จะทำให้ฐานข้อมูลปลอดภัยที่สุดเพราะผูกสิทธิ์เข้ากับ Database ตรงๆ แต่จะกระทบโค้ดเยอะมาก (ถือเป็นการ Refactor ใหญ่)
