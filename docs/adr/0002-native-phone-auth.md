# ADR 0002: การใช้ Native Phone + Password Auth แทนการใช้ Fake Email

**Date:** 2026-06-24

## สถานะ (Status)
**Accepted**

## บริบท (Context)
แต่เดิมการตั้งค่าระบบ Supabase Auth ของ LanFlow อาศัยการสร้าง "อีเมลจำลอง" (Fake Email: `@phone.lanflow.invalid`) ในการผูกเข้ากับระบบยืนยันตัวตน เนื่องจาก Supabase Auth เริ่มต้นมาจากการรองรับการเข้าสู่ระบบผ่านอีเมลหรือบัญชีโซเชียลเป็นหลัก

อย่างไรก็ตาม ผู้ใช้บางส่วนไม่มีอีเมล หรือมีการใช้อีเมลเดียวร่วมกันหลายบัญชี ทำให้ไม่สามารถใช้อีเมลจริงมาผูกกับบัญชีแต่ละสาขาได้อย่างสะดวก และเมื่อมีการวิเคราะห์เพิ่มเติม พบว่าการแปลงไปมาระหว่างเบอร์โทรกับอีเมลจำลอง สร้างความซับซ้อนให้กับ:
- การ Query ข้อมูล
- การอ่าน Audit Logs
- การทำงานในฝั่ง Admin (Provisioning)

## การตัดสินใจ (Decision)
เราตัดสินใจ **ยกเลิกการใช้อีเมลจำลองทั้งหมด** และเปลี่ยนมาใช้ความสามารถ **Phone Provider ของ Supabase Auth แบบ Native ร่วมกับรหัสผ่าน** แทน โดย:

1. หน้าต่างเข้าสู่ระบบ (Login) จะส่งคำสั่ง `supabase.auth.signInWithPassword({ phone, password })` 
2. ข้อมูลเบอร์โทรศัพท์ที่พนักงานกรอก (เช่น `0800000001`) จะถูกแปลงให้อยู่ในมาตรฐาน E.164 (`+66800000001`) ตั้งแต่ฝั่ง Frontend หรือฟังก์ชัน Middleware ก่อนถูกส่งไปยัง Supabase
3. เปิดระบบ Provider `Phone` ใน Supabase Dashboard โดยไม่ต้องใช้หรือตั้งค่า SMS Provider ใดๆ
4. สคริปต์สำหรับการนำเข้าข้อมูล (Migration Scripts) ของบัญชีพนักงานเดิม จะถูกอัปเดตให้ผูกบัญชีด้วย `phone` ไปยังตาราง `auth.users` โดยตรง 

## ผลที่ตามมา (Consequences)
**ข้อดี (Pros):**
- เข้าใจง่าย: 1 บัญชีผู้ใช้ต่อ 1 เบอร์โทรศัพท์โดยไม่มีการดัดแปลงเบื้องหลังให้สับสน
- ลดโอกาสเกิดข้อผิดพลาดในการทำ Migration และดูแลรักษาระบบ (Maintenance)
- การระบุตัวตน (Identity) ชัดเจนและสอดคล้องกับพฤติกรรมผู้ใช้ในประเทศไทย

**ข้อจำกัดที่ต้องยอมรับ (Cons / Trade-offs):**
- ฟังก์ชันจัดการลืมรหัสผ่าน (Forgot Password) จะต้องให้ Admin รีเซ็ตให้ หรือต้องมีการลงทุนซื้อ API SMS เพิ่มเติมในอนาคต หากต้องการให้พนักงานกดรับรหัสใหม่ด้วยตัวเองทางโทรศัพท์ (OTP)
- รูปแบบเบอร์โทรจะต้องถูกตรวจสอบความถูกต้อง (Validation & Normalization) และแปลงเป็น E.164 ทุกครั้งก่อนเรียกใช้ Supabase Auth API
