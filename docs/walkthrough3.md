# สรุปผลการปรับปรุงฟีเจอร์รายรับ-รายจ่าย (Income/Expense Sales Bill Adjustment)

ผมได้ดำเนินการปรับปรุงฟีเจอร์รายรับ-รายจ่าย ตามเอกสารแผนงานที่ระบุไว้เรียบร้อยแล้ว โดยมีรายละเอียดการแก้ไขดังนี้ครับ:

## สิ่งที่ได้รับการแก้ไขและเพิ่มเติม

### 1. Database Migration & Schema
- สร้างไฟล์ migration `20260702000000_income_sale_items.sql`
- สร้างตาราง `income_sale_items` เพื่อเก็บรายการบิลขาย (Global Catalog) พร้อมระบบจัดการ RLS สำหรับสิทธิ์การใช้งาน
- เพิ่มข้อมูลเริ่มต้น (Seed data) ได้แก่ `น้ำกรดตราเสือไฟท์` และ `น้ำกรดตรามังกรไฟท์`
- Migrate ข้อมูลเดิมจากตาราง `income_expense`:
  - อัปเดต `บิลน้ำกรด` เป็น `บิลขาย`
  - ลบหรือซ่อน (Soft delete) ข้อมูลตัวเลือกที่ถูกยกเลิก (`บิลทั่วไป`, `บิลค่าแรง`, `สูญหาย`)
  - Drop column `transaction_option` ทิ้งตามแผน
- อัปเดต `supabase-schema.sql` ให้สอดคล้องกับ Migration ใหม่

### 2. TypeScript Types & Data Hooks
- เพิ่มและอัปเดต Type ใน `src/types/index.ts` ได้แก่ `IncomeBillOption`, `ExpenseBillOption`, และ `IncomeSaleItem`
- สร้าง hook ใหม่ `useIncomeSaleItems` สำหรับดึงข้อมูลและจัดการรายการบิลขาย
- ปรับแต่ง hook `useIncomeExpense`
  - ถอดตัวแปร `transactionOption` ออกจากการบันทึกและระบบ Type ทั้งหมด
  - อัปเดต Logic เพื่อให้บันทึก `unit` และ `price` เฉพาะกรณีที่เป็น **บิลขาย** 
  - เพิ่มการส่งตัวแปร `created_by_user_id` เข้าไปในจังหวะ Insert บันทึกข้อมูล

### 3. Validation & Constrains
- **DB Level**: เพิ่ม `check constraint` เพื่อรับประกันว่า `bill_option` จะมีค่าที่ถูกต้องตามประเภท (`รายรับ`/`บิลขาย` สำหรับ Income, และ `ค่าใช้จ่าย` สำหรับ Expense) 
- **DB Level**: เพิ่ม `unique index` (แบบ partial) กันการใส่ชื่อรายการบิลขายที่เปิดใช้งานอยู่ซ้ำกัน
- **DB Level**: เพิ่ม `GRANT` ให้กับ Role `authenticated` ในสิทธิ์ `select, insert, update` ตารางใหม่
- **UI Level**: ปรับเงื่อนไขการบันทึกใน `LanFlowApp.tsx` (handleSubmit) ให้ห้ามบันทึกถ้ายอดเงินเท่ากับ 0 
  - หากเป็น **บิลขาย**: `unit > 0` และ `price > 0`
  - หากเป็น **รายรับ/ค่าใช้จ่าย**: `cost > 0`

### 4. UI: ฟอร์ม IncomeExpenseModal
- ลบส่วน **"ช่องทางการรับจ่ายเงิน"** ออกจากหน้าต่างรับ-จ่าย
- ปรับตัวเลือก **"รูปแบบ"** ของรายรับ ให้เหลือเพียง `รายรับ` และ `บิลขาย` (ส่วนรายจ่ายเหลือเพียง `ค่าใช้จ่าย`)
- ทำ Dynamic Form Branching ตามประเภทที่เลือก:
  - หากเป็น **บิลขาย**: ให้เลือกรายการจาก Dropdown, ระบุจำนวน, ราคา และแสดงผลรวมอัตโนมัติ (คำนวณจาก จำนวน x ราคา)
  - หากเป็น **รายรับ/ค่าใช้จ่าย**: จะยุบช่องจำนวนและราคาออกไป ให้กรอกเฉพาะชื่อรายการและจำนวนเงินสุทธิแทน

### 5. UI: หน้าต่างจัดการบิลขาย (IncomeSaleItemsModal)
- สร้างหน้าต่าง `IncomeSaleItemsModal` สำหรับให้ระดับ `super_admin` สามารถเพิ่มรายการบิลขายใหม่ๆ ได้ และสามารถปิดใช้งาน (disable) รายการเก่าที่ไม่ต้องการได้
- ดึงรายการทั้งหมดแสดงใน Modal (ทั้งที่ใช้งานและปิดใช้งานแล้ว) เพื่อให้เห็นประวัติทั้งหมด แต่ส่วนหน้าจอรับ-จ่ายจะเห็นแค่รายการที่ยัง Active
- เพิ่มปุ่ม **"เพิ่มรายการบิลขาย"** ลงในส่วนหน้าควบคุม (Action Bar) โดยจะแสดงขึ้นมาเฉพาะผู้ใช้งานระดับ `super_admin` เท่านั้น

## การตรวจสอบความถูกต้อง (Verification)
ผมได้รันคำสั่งตรวจสอบเพื่อยืนยันว่าการแก้ไขทั้งหมดจะไม่ทำให้ระบบพัง:
- รันคำสั่ง `npx tsc --noEmit` เพื่อเช็ค Type ทั้งหมด (ผ่านเรียบร้อย)
- รันคำสั่ง `npm run build` เพื่อจำลองขั้นตอนการสร้าง Production Build (ผ่านเรียบร้อย)
